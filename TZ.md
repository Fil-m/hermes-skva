1. ЗАГАЛЬНИЙ ОПИС ТА ПРИЗНАЧЕННЯ
══════════════════════════════════════════════════════════════════

СКВА — це набір скілів, інструментів, промптів і протоколів,
які дозволяють одному Hermes Agent (Оркестратор) створювати
колектив агентів-помічників для автоматизованого виробництва
програмних продуктів.

Система не потребує серверів, баз даних, WebSocket, Redis,
Docker чи іншої інфраструктури. Швидкі сигнали (heartbeat,
gate) — через локальну файлову систему або SSH. Артефакти
(код, spec) — через Git. Агенти запускаються на різних
машинах через SSH.

┌─────────────────────────────────────────────────────────────┐
│ ОСНОВНІ ХАРАКТЕРИСТИКИ                                      │
├─────────────────────────────────────────────────────────────┤
│ Платформа         │ Hermes Agent (CLI)                      │
│ Комунікація       │ Файли + Git (жодних прямих чатів)       │
│ Транспорт         │ SSH + terminal(background)              │
│ Інтерфейс         │ Hermes CLI + Telegram/Discord Gateway   │
│ Режими            │ Автоматичний + Налаштування             │
│ Методи            │ Solo, Rada+Fabryka, Agile Team, Pipeline│
│                   │ (відповідають сучасним методологіям:     │
│                   │  Waterfall/Lean, Waterfall-Agile, Scrum, │
│                   │  Continuous Delivery)                   │
│ Бюджетування      │ Вибір моделі за вартістю + ліміт проекту│
│ Навчання          │ Збір досвіду → створення скілів         │
└─────────────────────────────────────────────────────────────┘

Мета: дати користувачу робочий продукт, максимально наближений
до запиту, повністю приховуючи рутину виробництва. Користувач
бачить прогрес і може впливати, але не займається технічними
деталями.

2.2. Технологічний стек

| Компонент         | Роль                           | Примітка                            |
|-------------------|--------------------------------|-------------------------------------|
| Hermes Agent      | Базовий AI-агент               | CLI, інструменти: terminal, file,...|
| terminal tool     | Спавн агентів                  | background=true + SSH backend       |
| SSH               | Віддалений запуск              | Через terminal(backend=ssh)         |
| Git               | Синхронізація артефактів       | Push/pull, робота в ізоляції        |
| skill_view()      | Завантаження скілів агентами   | SKILL.md файли                      |
| skill_manage()    | Створення/редагування скілів   | З успішних патернів                 |
| memory()          | Міжсессійне навчання           | Факти про користувача, проекти      |
| session_search()  | Контекст з минулих сессій      | FTS5 пошук                          |
| cronjob()         | Регулярні/відкладені задачі    | Nightly build, запланований деплой  |
| delegate_task()   | Внутрішні підзадачі агента     | Не замість spawn агентів            |
| Gateway           | Нотифікації                    | Telegram / Discord                  |
| Profiles          | Постійні ролі                  | Окремі конфіги для Agile Team       |

2.3. Що НЕ використовується

❌ Веб-сервер (ніякого HTTP API)
❌ База даних (ніякої SQL/NoSQL)
❌ WebSocket (ніякого real-time)
❌ Redis / RabbitMQ / ZeroMQ (ніяких черг)
❌ Docker / Kubernetes
❌ Node.js / React / Vue (для UI)
❌ Зовнішні сервіси (крім GitHub для Git)

Уся комунікація між агентами — через файлову систему.
Git використовується ТІЛЬКИ для артефактів (код, spec, docs),
де потрібна історія змін та синхронізація між машинами.

3. ІЄРАРХІЯ СИСТЕМИ
══════════════════════════════════════════════════════════════════

Method → Phase → Role → Skill → Agent

Method (Метод виробництва)
  └── Визначає: які фази, які ролі, часові ліміти, модель
  └── Приклад: Rada+Fabryka, Solo, Agile Team, Pipeline

Phase (Фаза)
  └── Визначає: вхідний gate, набір ролей, вихідні артефакти
  └── Приклад: Council, Factory, Deploy, Sprint

Role (Роль / Професія агента)
  └── Визначає: набір скілів, модель за замовчуванням,
  │             personality, транспорт
  └── Приклад: Architect, Developer, QA, DevOps

Skill (Скіл)
  └── SKILL.md файл з інструкцією для агента
  └── Завантажується через skill_view()
  └── Приклад: react-coding, system-design, testing

Agent (Екземпляр)
  └── Конкретний запущений Hermes з роллю + скілами
  └── Має: session_id, машину, PID, статус, timeout

Принципи ізоляції:

1. Кожен агент пише ТІЛЬКИ в artifacts/<role>/ — ніколи не чіпає
   файли інших агентів або src/

2. Оркестратор збирає фінальний результат після gate: копіює
   artifacts/factory/src/ → src/

3. Lock-файли в signals/locks/ запобігають race conditions

4. Git push після кожного артефакту (не чекати кінця фази)

5. Конфліктів Git немає, бо агенти не редагують спільні файли.
   Якщо конфлікт виник (наприклад, два Dev пишуть один файл) —
   запускається Merge Agent (окремий Hermes з роллю Merger).

5. AGENT REGISTRY
══════════════════════════════════════════════════════════════════

Реєстр машин, на яких можна запускати агентів.

```yaml
# .hermes/agents/registry.yaml
machines:
  - name: wsl-local
    transport: terminal          # Локальний spawn
    host: localhost
    max_agents: 3
    roles:
      - Fullstack                # Solo метод
      - Developer                # Rada+Fabryka
      - QA
    models:
      - deepseek-flash
      - deepseek
      - qwen3-235b

  - name: hetzner-prod
    transport: ssh               # Віддалений spawn
    host: 116.203.217.173
    user: root
    key: /mnt/c/Users/TRS/.ssh/id_rsa
    max_agents: 2
    roles:
      - Architect
      - DevOps
    models:
      - qwen3-235b               # Тільки велика модель

  - name: future-machine         # Заглушка для майбутнього
    transport: ssh
    host: 192.168.1.100
    user: dev
    key: ~/.ssh/future_key
    max_agents: 1
    roles:
      - Developer
    models:
      - deepseek
```

Правила:

1. Оркестратор читає registry.yaml перед кожним spawn
2. Якщо машина недоступна — обирає наступну з registry
3. Якщо всі машини недоступні — агент вважається crashed
4. max_agents запобігає перевантаженню машини
5. Registry редагується через `/registry` в режимі Налаштування

  │                │ (гібрид)            │ Потім виробництво (Фабрика)  │
  │                │                     │ Деплой після review          │
  │                │                     │ Час: 60 хв                  │
  │                │                     │ Повне ТЗ перед кодом         │

  Extreme Programming, Kanban) через створення методу в
  режимі Налаштування: /method create --from Solo

6.1. Система продуктивної суперечки (Pairs + Judge)

  Ключовий механізм якості: агенти не просто співпрацюють,
  а СПЕРЕЧАЮТЬСЯ. Кожна пара має природний конфлікт інтересів,
  що породжує глибше опрацювання рішень.

  ПАРИ З ПРОТИЛЕЖНИМИ ІНТЕРЕСАМИ:

  │ Пара                      │ Що захищає                    │ У чому конфлікт              │
  │───────────────────────────┼───────────────────────────────┼──────────────────────────────│
  │ Architect vs DevOps       │ Інновації vs Стабільність     │ Architect: "React, новий     │
  │                           │                               │ стек!" DevOps: "Vanilla JS,  │
  │                           │                               │ перевірено роками"           │
  │ Analyst vs Developer      │ Об'єм vs Простота             │ Analyst: "всі фічі!"         │
  │                           │                               │ Developer: "мінімум для MVP" │
  │ Designer vs QA            │ Краса vs Надійність           │ Designer: "темна тема!"      │
  │                           │                               │ QA: "щоб не падало"          │
  │ Product Owner vs Security │ Швидкість vs Безпека          │ PO: "деплой сьогодні!"       │
  │                           │                               │ Security: "а де SSL?"        │

  ЯК ЦЕ ПРАЦЮЄ В РАДІ:

  ┌──────────────────────────────────────────────────────────────┐
  │ Тема: "Який стек обрати для тетрісу?"                       │
  │                                                              │
  │ Architect: "Canvas 2D — гнучкість, 60fps, control"          │
  │ DevOps:    "DOM — простіше, менше коду, швидше деплой"     │
  │                                                              │
  │ Раунд 1: Architect пише свою позицію                         │
  │ Раунд 2: DevOps читає, спростовує                             │
  │   "Canvas потребує ручного game loop, DOM все робить сам"    │
  │ Раунд 3: Architect відповідає                                 │
  │   "DOM не потягне 60fps при 20+ фігурах на екрані"           │
  │                                                              │
  │ Mentor (суддя):                                              │
  │   "Canvas 2D. Причина: Tetris має 7 фігур × 4 ротації =     │
  │    28 станів, DOM не оптимізований для частої         │
  │    перемальовки. DevOps має рацію щодо складності,           │
  │    але game loop на requestAnimationFrame вирішує це."       │
  └──────────────────────────────────────────────────────────────┘

  ПРАВИЛА СУПЕРЕЧКИ:

  1. Кожен аргумент має бути ФАКТИЧНИМ:
     ✅ "Canvas 2D забезпечує 60fps"
     ❌ "Canvas краще" (без обґрунтування)

  2. Кожен може спростувати аргумент іншого:
     "Твердження: 'Canvas складніший' — це недолік,
      але він компенсується продуктивністю"

  3. Суддя (Mentor) приймає рішення на основі:
     - Фактів, не думок
     - Відповідності обмеженням проекту
     - Балансу між аргументами пар

  4. Якщо пара не дійшла згоди за 2 раунди:
     - Mentor приймає рішення
     - Програша сторона отримує "право вето" на наступне
       питання (компенсація)

  5. Результат суперечки фіксується:
     .hermes/discussion/{topic}/resolution.md
     Містить: хто що пропонував, чому переміг інший варіант

  ПАРИ В РІЗНИХ ФАЗАХ:

  │ Фаза       │ Пара 1              │ Пара 2                │ Суддя          │
  │────────────┼─────────────────────┼───────────────────────┼────────────────│
  │ Council    │ Architect vs DevOps │ Analyst vs Developer  │ Mentor         │
  │ Factory    │ Developer vs QA     │ —                     │ Оркестратор    │
  │ Review     │ QA vs Developer     │ —                     │ Mentor         │
  │ Sprint     │ PO vs Team          │ Security vs Developer │ Scrum Master   │
  │ Retro      │ Team vs Process     │ —                     │ Retro Agent    │

  МЕТА: кожне рішення проходить через суперечку, що гарантує
  якість. Без суперечки — рішення приймається "на віру", що
  в AI-агентів призводить до шаблонних помилок.

6.2. SOLO — "зроби швидко"

Призначення: дрібні задачі, фікси, однофайлові зміни.

┌─────────────────────────────────────────────────────────────┐
│ Характеристика        │ Значення                             │
│───────────────────────┼─────────────────────────────────────│
│ Модель                │ DeepSeek Flash (найдешевша)          │
│ Timeout               │ 15 хвилин                            │
│ Агентів               │ 1 (Fullstack)                        │
│ Фази                  │ Execute → Verify → Deliver           │
│ Спосіб spawn          │ terminal(background)                 │
└─────────────────────────────────────────────────────────────┘

Flow:
  1. Оркестратор аналізує: запит < 100 слів, однотипний
  2. Spawn: terminal(bg, "hermes chat -q '{role_prompt}'")
     де role_prompt містить:
       - роль (Fullstack)
       - скіли (react-coding, bug-fixing)
       - задачу (оригінальний запит)
       - інструкцію: писати в .hermes/artifacts/
       - heartbeat кожні 60с
       - сигнал .done при завершенні
  3. Оркестратор чекає .done або timeout
  4. Верифікує: код компілюється / тести проходять
  5. Доставка: посилання/патч

Приклад:
  Користувач: "пофікси баг — кнопка Save не працює"
  Оркестратор → Solo → Fullstack → fix → verify → "готово"

6.3. RADA+FABRYKA — "спроектуй та зроби"

Призначення: нові проекти, складні задачі, де потрібна архітектура.

┌─────────────────────────────────────────────────────────────┐
│ Характеристика        │ Значення                             │
│───────────────────────┼─────────────────────────────────────│
│ Модель Council        │ DeepSeek (дешева для тексту)         │
│ Модель Factory        │ Qwen3-235B через Gonka (код)         │
│ Timeout               │ 60 хвилин                            │
│ Агентів               │ 4-5 (Architect, Analyst, DevOps,      │
│                       │        Mentor, Developer, QA)          │
│ Фази                  │ Council → Factory → Review → Deploy  │
│ Спосіб                │ terminal(bg) + SSH                   │
└─────────────────────────────────────────────────────────────┘

ФАЗА 1: COUNCIL (Рада) — створення ТЗ

  Рада — це **повноцінна багатоагентна дискусія**, аналогічна
  тому як ми з тобою створювали це ТЗ: раунди, заперечення,
  консенсус, ітерації.

  Учасники:
    - Architect (архітектура, стек, компоненти)
    - Analyst (вимоги, ризики, обмеження)
    - DevOps (деплой, інфраструктура)
    - Mentor (рев'ю якості ТЗ, перевірка несуперечностей)
    Всі — окремі Hermes сессії, spawn через terminal(background)

  Тривалість: до 30 хв (5 раундів по 5 хв)
  Модель: DeepSeek (дешева, але достатня для тексту)
  Формат: файл-форум (секція 9.6, режим "Складна дискусія")

  Процес:

  РАУНД 1 — Початкові позиції:
    Кожен учасник spawn-иться, читає user-request.md, пише:
    - architect-position.md: "Стек: Vanilla JS + Canvas. Чому..."
    - analyst-position.md: "Вимоги: X, Y, Z. Ризики: A, B"
    - devops-position.md: "Деплой: GitHub Pages. Обмеження: ..."

  РАУНД 2 — Реакція:
    Кожен читає позиції інших, оновлює свою:
    - "Architect: analyst має рацію щодо Canvas, але...
       Пропоную компроміс:..."
    - "Analyst: згоден з devops, прибираю обмеження..."

  РАУНД 3-5 — Консенсус (або суперечка):
    - Якщо всі згодні → Mentor пише consensus.md
    - Якщо суперечка → додаткові раунди
    - Якщо 5 раундів без консенсусу → Mentor приймає рішення
    - Mentor перевіряє: чи несуперечливе ТЗ? чи всі вимоги
      покриті? чи є критерії приймання?

  Результат Ради — повноцінне ТЗ:

    .hermes/artifacts/council/
    ├── tz.md                          # Повне ТЗ
    ├── architect-position.md          # Позиція Architect
    ├── analyst-position.md            # Позиція Analyst
    ├── devops-position.md             # Позиція DevOps
    ├── mentor-review.md               # Рев'ю Mentor
    ├── discussion-log.md              # Лог всіх раундів
    └── consensus.md                   # Фінальне рішення

  Gate: tz.md існує + mentor-review.md пройшов

  Детальна методологія створення ТЗ описана в секції 15.3.
  Council учасники отримують її як інструкцію при spawn.

ФАЗА 2: FACTORY (Фабрика)

  Учасники: Developer, QA
  Час: до 30 хв
  Модель: Qwen3-235B (найкраща для коду)

  Developer:
    1. Читає artifacts/council/spec.md + arch.md
    2. Створює src/ з кодом
    3. Встановлює залежності
    4. Тестує: npm test / python -m pytest
    5. Сигнал: .factory.done

  QA:
    1. Читає код з artifacts/factory/src/
    2. Доповнює тести
    3. Запускає тести
    4. Якщо тести впали — пише баг-репорт
    5. Сигнал: .qa.done

  Оркестратор:
    - Обидва сигнали? → верифікація
    - Build проходить? → gate: factory.verified
    - Build не проходить? → сигнал .factory.fail → retry

ФАЗА 3: REVIEW

  Учасник: Оркестратор або окремий Reviewer агент
  Час: до 5 хв

  - Перевіряє код на:
    - Відповідність spec
    - Безпеку (ніяких секретів у коді)
    - Стиль (ніякого мотлоху)
  - Якщо ок → gate: review.passed
  - Якщо ні → список правок → Developer retry

ФАЗА 4: DEPLOY

  Учасник: DevOps
  Час: до 10 хв
  Модель: DeepSeek Flash

  - GitHub Pages: npm run build → gh-pages
  - Або: копіює build/ в деплой-директорію
  - Пише url.txt: https://user.github.io/project
  - Сигнал: .deploy.done

  Оркестратор: curl url → 200? → gate: deploy.verified

6.4. AGILE TEAM — "великий проект з ітераціями"

Призначення: довготривалі проекти, де потрібен розвиток.

┌─────────────────────────────────────────────────────────────┐
│ Характеристика        │ Значення                             │
│───────────────────────┼─────────────────────────────────────│
│ Модель Dev            │ Qwen3-235B                           │
│ Модель QA/Review      │ DeepSeek                             │
│ Timeout               │ 2 години на спринт                   │
│ Агентів               │ 3-5 (постійні профілі)               │
│ Фази                  │ Planning → Sprint×N → Demo → Retro   │
│ Спосіб                │ Hermes Profiles                      │
└─────────────────────────────────────────────────────────────┘

Ролі:
  - Product Owner: Оркестратор (Hermes #1)
  - Scrum Master: Оркестратор
  - Developers: 2-3 профілі (frontend, backend)
  - QA: 1 профіль

Профілі Hermes (налаштовуються один раз):
  ~/.hermes/profiles/frontend/config.yaml:
    model: qwen3-235b
    skills: [react-coding, css, api-client]
    toolsets: [terminal, file]

  ~/.hermes/profiles/qa/config.yaml:
    model: deepseek
    skills: [testing, bug-report, e2e]
    toolsets: [terminal, file, web]

Процес спринту:
  1. Planning: PO формує backlog → команда оцінює → спринт N
  2. Sprint: агенти працюють, Оркестратор daily standup кожні 30м
     (перевіряє сигнали, heartbeat, вирішує блокери)
  3. Кінець: QA регресія → демо користувачу
  4. Retro: агенти пишуть що пішло не так

Артефакти спринту:
  .hermes/phases/history/sprint-N/backlog.yaml
  .hermes/phases/history/sprint-N/standup-logs/
  .hermes/phases/history/sprint-N/demo.md
  .hermes/phases/history/sprint-N/retro.md

6.5. PIPELINE — "конвеєр"

Призначення: CI/CD, однотипні задачі, де важлива послідовність.

┌─────────────────────────────────────────────────────────────┐
│ Характеристика        │ Значення                             │
│───────────────────────┼─────────────────────────────────────│
│ Модель                │ DeepSeek (кожен етап)                │
│ Timeout               │ 45 хвилин на весь конвеєр            │
│ Агентів               │ 4 (по одному на фазу, послідовно)    │
│ Фази                  │ Spec → Code → Test → Deploy          │
│ Спосіб                │ terminal(bg), послідовний spawn      │
└─────────────────────────────────────────────────────────────┘

Flow:

  Agent Spec  ──→ .spec.done ──→ Agent Code ──→ .code.done
       ↓                              ↓
  spec.yaml                     src/ + tests/

  Agent Test  ──→ .test.done ──→ Agent Deploy ──→ .deploy.done
       ↓                              ↓
  test report                    deployed URL

Кожен наступний агент spawn-иться тільки після gate попереднього.
Якщо фаза failed — pipeline stops, Оркестратор вирішує: retry,
заміна агента, або abort з повідомленням.

══════════════════════════════════════════════════════════════════
7. ОРКЕСТРАТОР
══════════════════════════════════════════════════════════════════

  Оркестратор — це той самий Hermes, в якому працює користувач.
  Він не створює окремий процес для себе.

  Sequence diagram типового проекту (Rada+Fabryka):

  ФАЗА 0: ІДЕАЦІЯ (Brain Dump) — збір ідеї від користувача

  До того як запускати Council, Оркестратор проходить з
  користувачем фазу збору ідеї. Без цього ТЗ буде неповним.

  Процес:

  ┌─────────────────────────────────────────────────────────────┐
  │ Користувач: "хочу тетріс"                                   │
  │                                                             │
  │ Оркестратор пропускає ідею через ШІ:                        │
  │ → "Тетріс: класична гра-головоломка. 7 фігур, 10×20 поле.  │
  │    Типові вимоги: темна тема, рекорди, пауза."              │
  │                                                             │
  │ Оркестратор: "Додати ці базові вимоги? [Y/n]"              │
  │ Користувач: "Y, і додай звук"                               │
  │                                                             │
  │ Оркестратор: "Опиши 2-3 сценарії використання:"             │
  │ Користувач: "як гравець я хочу бачити рахунок.              │
  │  як гравець я хочу паузу. як гравець я хочу темну тему."    │
  │                                                             │
  │ Оркестратор аналізує достатність:                           │
  │ ✓ Вимоги: гра, темна тема, рекорди, пауза, звук            │
  │ ✓ User stories: 3 шт                                        │
  │ ✓ Обмеження: GitHub Pages, offline                          │
  │ → Достатньо для Council? ✅                                  │
  │                                                             │
  │ Оркестратор: "ТЗ достатньо вичерпне. Запускаю Раду?"       │
  │ Користувач: "так"                                           │
  │ → Council                                                  │
  └─────────────────────────────────────────────────────────────┘

  Правила фази Ідеації:

  1. Користувач дає МІНІМУМ — одне речення
  2. Оркестратор розширює через ШІ (що типічно для такого проекту)
  3. Користувач підтверджує/відхиляє/доповнює
  4. Користувач дає 2-3 user stories (вільним текстом)
  5. Оркестратор перевіряє достатність за Spec Quality Framework
  6. Якщо не вистачає — задає уточнюючі питання
  7. Максимум 5 раундів уточнень
  8. Після 5 раундів або підтвердження → Council

  Критерії достатності (всі мають бути ✅):

  │ Критерій                     │ Перевірка                        │
  │──────────────────────────────┼─────────────────────────────────│
  │ Зрозумілий тип проекту       │ "гра", "сайт", "застосунок"     │
  │ Мінімум 3 вимоги             │ функціональні + нефункціональні │
  │ Мінімум 2 user stories       │ "як гравець, я хочу..."         │
  │ Зрозумілі обмеження          │ платформа, стек, дедлайн        │
  │ Користувач підтвердив        │ "так", "давай", "ок"            │

  User        Orchestrator    Architect  Analyst   DevOps   Mentor  Developer  QA   Deploy
  │               │             │          │         │       │       │         │      │

7.1. Балансування виробництва під ресурси (Resource-Aware)

  Система автоматично підлаштовується під потужність комп'ютера.
  Слабкий — один агент за раз. Потужний — паралельні агенти.

  ЯК ВИЗНАЧАЄТЬСЯ ПОТУЖНІСТЬ:

  При старті Оркестратор визначає можливості:

  │ Ресурс              │ Як вимірюється                    │ Вплив                        │
  │─────────────────────┼──────────────────────────────────┼──────────────────────────────│
  │ CPU                 │ nproc (кількість ядер)            │ max_parallel_agents          │
  │ RAM                 │ free -m (вільна пам'ять)          │ max_agents_per_machine       │
  │ Load average        │ uptime (1 min, 5 min, 15 min)    │ Поточне навантаження         │
  │ Мережа (SSH)        │ ping RTT (туди-назад)            │ Чи варто spawn на віддалені  │

  Правила:

  │ Потужність          │ Умова            │ max_agents │ Приклад                       │
  │─────────────────────┼──────────────────┼────────────┼───────────────────────────────│
  │ Слабкий             │ 2 ядра, <4GB RAM │ 1          │ Ноутбук, Raspberry Pi         │
  │                     │                  │            │ Повільно, але працює          │
  │ Середній            │ 4 ядра, 8GB RAM  │ 2-3        │ Звичайний ПК                  │
  │                     │                  │            │ Developer + QA паралельно     │
  │ Потужний            │ 8+ ядер, 16+GB   │ 4-6        │ Робоча станція, сервер        │
  │                     │                  │            │ Council + Factory + QA + Deploy│
  │ Кластер             │ Декілька машин   │ Сума всіх  │ WSL + Hetzner + друга машина  │
  │                     │ в registry       │            │ Агенти розподілені            │

  АЛГОРИТМ БАЛАНСУВАННЯ:

  ```python
  def calculate_capacity(machine):
      """Визначити скільки агентів може працювати паралельно."""

      # 1. Фізичні ресурси
      cpu_cores = get_cpu_cores(machine)        # nproc
      free_ram = get_free_ram(machine, "gb")     # free -m

      # 2. Поточне навантаження
      load_avg = get_load_average(machine)       # uptime
      active_agents = count_active_agents(machine)

      # 3. Розрахунок
      # Кожен агент ≈ 1 ядро + 2GB RAM
      from_cores = cpu_cores - load_avg          # вільні ядра
      from_ram = free_ram // 2                   # скільки агентів по 2GB
      from_registry = machine.max_agents         # ліміт з конфігу

      # 4. Мінімум з усіх лімітів
      max_new = min(from_cores, from_ram, from_registry)
      max_new = max(max_new, 1)  # хоча б 1 агент

      return max_new
  ```

  ЯК ЦЕ ПРАЦЮЄ НА ПРАКТИЦІ:

  Слабкий ноутбук (2 ядра, 4GB RAM):
    → Capacities: 1 агент
    → Council: Architect по черзі → Analyst по черзі (не паралельно)
    → Factory: Developer один
    → Час: довше, але працює
    → Користувач бачить: "Ваш комп'ютер слабкий, проект буде
      готовий за ~10 хв замість 5 хв"

  Потужний ПК (8 ядер, 32GB RAM):
    → Capacities: 4-6 агентів
    → Council: всі 4 агенти паралельно (Architect, Analyst, DevOps, Mentor)
    → Factory: Developer + QA паралельно
    → Час: мінімальний
    → Користувач бачить: "Використовую всю потужність, проект за 3 хв"

  Кластер (WSL + Hetzner):
    → Capacities: сума всіх машин
    → Розподіл: Council на локальному, Factory на Hetzner
    → Час: оптимальний

  Оркестратор динамічно змінює parallelism під час проекту:
  - Якщо load average зріс >80% → зменшити кількість агентів
  - Якщо агент timeout через нестачу ресурсів → зменшити parallelism
  - Якщо з'явилась нова машина в registry → додати ресурси

7.4. Оптимізація виробництва (Production Optimization)

  Виправлення від продакшн-менеджера — усувають простої,
  великі партії та відсутність пріоритетів.

  7.4.1. Інкрементальна доставка (thin slices)

    Замість "весь проект → деплой", Rada+Fabryka може
    доставляти частинами:

    Звичайний режим:
      Council(30м) → Factory(30м) → Deploy(10м) → результату

    Інкрементальний режим:
      Council: архітектура + 1-й екран → Factory: 1-й екран
        → Deploy: 1-й екран ✅ (5 хв — користувач бачить!)
      → Factory: 2-й екран → Deploy: 2-й екран ✅
      → Factory: 3-й екран → Deploy: готово ✅

    Критерій: якщо проект > 3 екранів або > 30 хв → thin slices.

    Приклад thin slices для "Калькулятор":

    Slice 1 — UI Skeleton + базові операції:
      Файли: index.html, styles.css, app.js
      Код: кнопки 0-9, +,-,×,÷, дисплей, обробка кліків
      Деплой: https://user.github.io/calc (версія 0.1)
      ⏱ 2 хв — користувач вже бачить інтерфейс!

    Slice 2 — логіка + помилки:
      Файли: calculator.js (додано evaluate())
      Код: обчислення, ділення на 0, синтаксичні помилки
      Деплой: та сама URL, оновлена версія
      ⏱ +1 хв — користувач може рахувати!

    Slice 3 — історія + теми:
      Файли: storage.js (localStorage), themes.css
      Код: збереження історії, темна/світла тема
      Деплой: фінальна версія
      ⏱ +1 хв — готово!

    Всього: 4 хв замість 15 хв (thin slices економія 73%)


  7.4.2. Перекриття фаз (pipeline overlap)

    Замість простою між фазами — перекриття:

    │ Час │ Council     │ Factory     │ QA          │ Deploy      │
    │─────┼─────────────┼─────────────┼─────────────┼─────────────│
    │ 0   │ Architect   │             │             │             │
    │     │ пише ТЗ     │             │             │             │
    │ 3   │ DevOps      │             │             │             │
    │     │ сперечається│             │             │             │
    │ 10  │ Mentor судить│ Developer  │             │             │
    │     │             │ пише 1-й   │             │             │
    │     │ Council done│ модуль     │             │             │
    │ 15  │             │ QA тестує  │             │             │
    │     │             │ 1-й модуль │             │             │
    │ 17  │             │ Developer  │             │ DevOps      │

    Замість 60 хв → 22 хв. Економія: 63%.

  7.4.3. Пріоритети задач (Queue)

    │ Пріоритет │ Тег        │ Дія                                         │
    │───────────┼────────────┼─────────────────────────────────────────────│
    │ P0        │ critical   │ Перервати поточний проект → Solo fix       │
    │           │            │ Після fix → відновити перерваний проект     │
    │ P1        │ high       │ Наступним після поточного проекту           │
    │ P2        │ normal     │ В порядку черги                             │
    │ P3        │ low        │ Коли всі вищі пріоритети виконано          │

    Користувач: "/priority P0 — сайт впав!"
    → Оркестратор: checkpoint поточного проекту → Solo fix
    → Після fix → resume проекту з checkpoint

  7.4.4. Continuous verification

    Агент запускає тести після КОЖНОГО значущого файлу,
    а не чекає .done:

    Developer пише Calculator.jsx → автоматично: npm test
    → Якщо fail → миттєве виправлення (не через 20 хв)
    → Якщо pass → наступний файл

    Економія: помилка виявляється за 30с, не за 30 хв.

  7.4.5. Lean документація

    Об'єм документації залежить від методу:

    │ Метод        │ Документація                                      │
    │──────────────┼───────────────────────────────────────────────────│
    │ Solo         │ Тільки сигнали (.done, heartbeat)                 │
    │              │ Жодних spec, arch, logs — тільки код              │
    │ Rada+Fabryka │ Повна: .hermes/project/ + ТЗ + logs              │
    │ Agile Team   │ Повна + sprint backlog + retro                    │
    │ Pipeline     │ Середня: spec + artifacts, без logs агентів       │

  7.4.6. Multi-project pipeline

    Якщо ресурси дозволяють, Оркестратор веде 2+ проекти:

    │ Проект │ Статус      │ Фаза       │ Пріоритет │ Машина  │
    │────────┼─────────────┼────────────┼───────────┼─────────│
    │ Tetris │ running     │ Factory    │ P1        │ WSL     │
    │ Calc   │ queued      │ —          │ P2        │ —       │
    │ Blog   │ checkpoint  │ Council    │ P0        │ Hetzner │
    │        │ (paused)    │            │           │         │

    Різні проекти на різних машинах.
    Різні методи для різних проектів.
    Оркестратор розподіляє ресурси між проектами.

```python
def spawn_agent(role, task, project_path, skills, model="deepseek-flash"):
    """Створити агента на локальній машині."""

Твоя роль: {role.name}
```

8.2. Віддалений spawn (SSH) — повний приклад

  ```python
  import json, shlex

  def escape(s):
      """Екранувати prompt для shell-in-shell передачі."""
      # Замінити одинарні лапки: ' → '\''
      # Загорнути в одинарні лапки для безпеки
      return shlex.quote(s)

  def spawn_ssh_agent(machine, role, task, remote_path):
      """Створити агента на віддаленій машині через SSH."""

      prompt = f"""
  Твоя роль: {role.name}
  Скіли: {role.skills}
  Проект: {remote_path}
  Задача: {task}

  Працюй в .hermes/artifacts/{role.name}/
  Heartbeat: .hermes/signals/heartbeat/{role.name}.live кожні 60с
  Коли готово — touch .hermes/signals/.{role.phase}.done
  Не чіпай файли інших агентів
  """

      # shell-in-shell: ssh → hermes chat -q
      # Використовуємо одинарні лапки для всього SSH
      # prompt всередині теж екрануємо
      safe_prompt = prompt.replace("'", "'\\''")

      ssh_cmd = (
          f"ssh -i {machine.key} -o ConnectTimeout=10 "
          f"{machine.user}@{machine.host} "
          f"'cd {remote_path} && "
          f"source ~/.hermes/.env && "
          f"hermes chat -q '{safe_prompt}'"
          f"'"
      )

      terminal(
          background=True,
          command=ssh_cmd,
          timeout=1800  # 30 хв
      )
  ```

9. КОМУНІКАЦІЙНИЙ ПРОТОКОЛ

Агенти НІКОЛИ не спілкуються напряму (через чат, API, WebSocket).
Вся комунікація — через файли. Але є 3 рівні залежно від
швидкості та розташування:

┌────────────────────────────────────────────────────────────┐
│ РІВЕНЬ 1: ЛОКАЛЬНА ФАЙЛОВА СИСТЕМА (миттєво, <1ms)       │
│                                                            │
│ Коли агенти на одній машині (WSL):                         │
│   - heartbeat: touch .hermes/signals/heartbeat/dev.live    │
│   - gate: test -f .hermes/signals/.factory.done            │
│   - status: echo '{"phase":"factory"}' > status.json      │
│   - lock: fcntl.flock(lock_file, LOCK_EX)                  │
│                                                            │
│ Жодного Git, жодної мережі — просто читання/запис файлів. │
│ Швидкість: наносекунди.                                    │
├────────────────────────────────────────────────────────────┤
│ РІВЕНЬ 2: SSH СИГНАЛИ (швидко, ~100-300ms)                │
│                                                            │
│ Коли агенти на РІЗНИХ машинах (WSL ↔ Hetzner):            │
│                                                            │
│ Heartbeat з Hetzner на WSL:                                │
│   Оркестратор сам опитує:                                  │
│   ssh hetzner 'cat .hermes/signals/heartbeat/dev.live'    │
│                                                            │
│ Gate з Hetzner на WSL:                                     │
│   Оркестратор перевіряє:                                   │
│   ssh hetzner 'test -f .hermes/signals/.deploy.done'      │
│   && echo "done"                                           │
│                                                            │
│ Статус:                                                    │
│   ssh hetzner 'cat .hermes/signals/status/dev.json'       │
│                                                            │
│ Агент на віддаленій машині НЕ має SSH доступу до           │
│ WSL/Оркестратора. Оркестратор сам опитує всі машини.      │
│                                                            │
│ Жодного Git, жодних push/pull. Просто SSH + touch/read.   │
│ Швидкість: ~100-300ms (накладні витрати SSH).              │
├────────────────────────────────────────────────────────────┤
│ РІВЕНЬ 3: GIT АРТЕФАКТИ (повільно, 1-5с)                 │
│                                                            │
│ Git використовується ТІЛЬКИ для:                          │
│   - Коду (src/, tests/, package.json)                      │
│   - Документів (spec.md, arch.md, summary.md)              │
│   - Всього, де ПОТРІБНА ІСТОРІЯ ЗМІН                      │

9.1. Сигнали (signal files)

  Формат .done та .fail файлів:

  ```yaml
  # .hermes/signals/.factory.done
  # ❗ Простий marker файл. Вмісту немає — тільки існування.
  # Типу: touch .hermes/signals/.factory.done
  ```

  ```json
  # .hermes/signals/.deploy.fail
  # ❗ Містить причину помилки.
  {
    "reason": "npm test failed: 2 tests failing",
    "agent": "developer",
    "timestamp": "2026-06-07T14:35:00Z",
    "phase": "factory",
    "retry_count": 1
  }
  ```

  Для складних випадків — JSON з деталями.
  Для простих — достатньо `touch` і `echo "reason" > .fail`.

9.2. Heartbeat протокол (з grace period)

1. Агент при старті:
   echo '{"ts":'$(date +%s)'}' > .hermes/signals/heartbeat/{role}.live

2. Агент кожні 60с оновлює timestamp:
   echo '{"ts":'$(date +%s)'}' > .hermes/signals/heartbeat/{role}.live

2. Оркестратор кожні 30с перевіряє всі .live файли:

   Локальні агенти (WSL):
     for role, live_file in list_heartbeats():
         data = json.load(open(live_file))
         age = now - data["ts"]

   Віддалені агенти (SSH):
     for machine in registry:
         for role in machine.active_agents:
             result = ssh(machine, "cat .hermes/signals/heartbeat/{role}.live")
             age = now - result["ts"]

   Агент на віддаленій машині НЕ має доступу до WSL.
   Оркестратор сам підключається до всіх машин і перевіряє.

```python
def check_heartbeats():
    now = time.time()
    for role, live_file in list_heartbeats():
        try:
            data = json.load(open(live_file))
            age = now - data["ts"]
            if age > 180:
                mark_crashed(role)
            elif age > 120:
                mark_warning(role, f"no heartbeat for {age:.0f}s")
        except:
            mark_crashed(role)
```

4. Grace period: 120-180с = WARNING, 180с+ = CRASHED
   (запобігає false positives при тимчасових затримках)

5. При CRASHED:
   - Спроба 1: retry на тій самій машині
   - Спроба 2: retry на іншій машині (з registry)
   - Спроба 3: retry з меншою моделлю (дешевше)
   - Після 3: REPLACED — новий агент з новою роллю

6. При перезавантаженні машини — всі heartbeats зникають →
   Оркестратор вважає всіх агентів на ній crashed

9.3. Gate протокол

```python
def check_gate(phase_name):
    done_file = f".hermes/signals/.{phase_name}.done"
    if not os.path.exists(done_file):
        return "PENDING"  # ще не готово

    fail_file = f".hermes/signals/.{phase_name}.fail"
    if os.path.exists(fail_file):
        reason = open(fail_file).read()
        return f"FAILED: {reason}"

    # Верифікація
    verified = verify_phase(phase_name)
    if not verified:
        return "VERIFICATION_FAILED"

    # Підвищити фазу
    next_phase = get_next_phase(phase_name)
    os.symlink(f"history/{next_phase}/", ".hermes/phases/current")
    return f"PASSED → {next_phase}"
```

  Recovery після .fail:
  При fail система автоматично:
  1. Логує: .hermes/logs/failures/{project}/{timestamp}.json
  2. Зберігає checkpoint: останній стабільний стан
  3. Сповіщає: Telegram "⚠️ Factory failed: npm test error"

  Користувач: hermes resume {project}
  → Оркестратор: відновлює стан з checkpoint
  → Продовжує з кроку де впало
  → Якщо 3 fail поспіль → статус blocked, ручний аналіз


9.4. Lock-файли (запобігання race conditions)

При одночасному доступі кількох агентів до спільних ресурсів
(наприклад, Git push):

```python
import fcntl

def with_lock(path, fn):
    with open(path + ".lock", 'w') as lock:
        fcntl.flock(lock, fcntl.LOCK_EX)  # exclusive lock
        try:
            result = fn()
        finally:
            fcntl.flock(lock, fcntl.LOCK_UN)
        return result
```

9.5. Протокол очікування залежностей (Polling)

Коли агент починає фазу, артефакти попередньої фази можуть
бути ще не готові. Агент не має падати — він має чекати.

Правило: перед початком роботи агент перевіряє наявність
вхідних артефактів кожні 15 секунд.

```python
def wait_for_artifact(path, timeout=300, interval=15):
    """Чекати появи файлу. timeout=None = без ліміту."""
    waited = 0
    while not os.path.exists(path):
        if timeout and waited >= timeout:
            return False  # timeout
        time.sleep(interval)
        waited += interval
    return True
```

│ Ситуація                          │ Чекає на              │ Polling │
│───────────────────────────────────┼───────────────────────┼─────────│
│ Developer починає Factory         │ .council.done          │ 15с     │
│                                   │ artifacts/council/     │         │
│                                   │   spec.md              │         │
│ QA починає тестування             │ .factory.done          │ 15с     │
│                                   │ artifacts/factory/src/ │         │
│ DevOps починає деплой             │ .review.passed         │ 15с     │
│                                   │ artifacts/factory/     │         │
│                                   │   build/               │         │
│ Агент в складній дискусії         │ .hermes/discussion/    │ 15с     │
│                                   │   .topic               │         │

Якщо timeout (наприклад, 5 хв) — агент сигналить .fail
з причиною "timeout waiting for {path}".

Для віддалених агентів (SSH): Оркестратор перевіряє наявність
артефактів на віддаленій машині через SSH перед spawn наступного
агента. Агенти на віддалених машинах не чекають — вони
отримують готовий контекст при spawn.

9.6. Групові дискусії (Council, Retro, Standup)

Для сценаріїв, де кілька агентів мають обговорити рішення,
використовується формат **багатоагентного файл-форуму**.
Кожен учасник — окрема Hermes сессія (не симуляція).

Файл-форум:

  .hermes/discussion/{topic-name}/
  ├── .topic                 # Питання: "Який стек обрати?"
  ├── round-1/
  │   ├── architect.md       # Початкова позиція
  │   ├── analyst.md
  │   └── devops.md
  ├── round-2/
  │   ├── architect.md       # Оновлена після прочитання інших
  │   ├── analyst.md
  │   └── devops.md
  ├── round-3/
  │   └── ...
  ├── mentor-review.md       # Перевірка якості
  ├── consensus.md           # Фінальне рішення
  └── .resolved              # Форум закрито

Процес:
  1. Оркестратор створює .topic з питанням
  2. Раунд N: spawn кожного учасника через terminal(background):
     - Читає .topic та всі попередні раунди
     - Пише/оновлює свою позицію
  3. Після кожного раунду Оркестратор перевіряє:
     - Чи всі написали?
     - Чи є консенсус?
  4. Якщо консенсус → Mentor пише consensus.md
  5. Якщо незгода через 5 раундів → Mentor вирішує
  6. .resolved → форум закрито

Учасники форуму:

│ Роль         | Що пише                         | Personality             |
│──────────────┼─────────────────────────────────┼─────────────────────────│
│ Architect    | Стек, архітектура, компоненти   | "консервативний,         |
│              |                                  | прагматичний"            |
│ Analyst      | Вимоги, ризики, обмеження       | "допитливий,             |
│              |                                  | скептичний"              |
│ DevOps       | Деплой, CI, інфраструктура      | "практичний,             |
│              |                                  | безпековий"              |
│ Mentor       | Рев'ю: несуперечності, якість   | "вимогливий,             |
│              | повнота, критерії               | педантичний"             |

ДОДАТКОВА РОЛЬ: Mentor

Mentor — це окремий агент, який НЕ бере участь в дискусії,
а перевіряє її результат:

  1. Чи всі позиції несуперечливі?
  2. Чи покриті всі вимоги з user-request.md?
  3. Чи є критерії приймання?
  4. Чи реалістичний план?
  5. Якщо ТЗ має проблеми → пише mentor-review.md зі списком
     ("Знайдені проблеми: 1. ..., 2. ...")

  Після виправлень → повторна перевірка
  Якщо Mentor схвалив → gate: council.done

Тривалість:
  - 3 раунди: ~10 хв (швидкий проект)
  - 5 раундів: ~25 хв (складний, суперечливий)

│ Тип артефакту  │ Що перевіряється                          │ Команда                             │
│────────────────┼───────────────────────────────────────────┼─────────────────────────────────────│
│ Код (JS/TS)    │ Компіляція без помилок                    │ `npx tsc --noEmit`                  │
│ Код (Python)   │ Синтаксис                                 │ `python -m py_compile src/main.py`  │

│ JSON/YAML      │ Валідний синтаксис                        │ `python -c "import yaml; yaml.safe_load(open('f'))"` │

10.2. Логіка верифікації

```python
def verify_phase(phase_name):
    config = {
        "factory": {
            "checks": [
                ("code_compiles", check_compile),
                ("tests_pass",    check_tests),
            ]
        },
        "deploy": {
            "checks": [
                ("url_responds", check_url),
            ]
        },
        "council": {
            "checks": [
                ("files_exist", check_artifacts_not_empty),
            ]
        }
    }

    for check_name, check_fn in config.get(phase_name, {}).get("checks", []):
        if not check_fn():
            log(f"Verification FAILED: {check_name}")
            return False

    log(f"Verification PASSED: {phase_name}")
    return True
```

10.3. При невдалій верифікації

1. Агент отримує 1 додаткову спробу (той самий spawn)
2. Якщо знову fail → retry з новим spawn (той самий prompt)
3. Якщо 2 retry failed → spawn з іншою моделлю (кращою)
4. Якщо все failed → метод вважається непридатним для цієї задачі
   → Оркестратор обирає інший метод або питає користувача

```python
def handle_feedback(user_text):
    """Аналіз відгуку користувача і вибір дії."""

    # Визначити рівень (простий keyword match + LLM аналіз)
    if user_text.lower() in ["так", "супер", "ok", "good", "прийнято"]:
        level = "ACCEPTED"
    elif any(w in user_text for w in ["не те", "неправильно", "не вірно"]):
        level = "WRONG_APPROACH"
    elif any(w in user_text for w in ["виправити", "додай", "зміни", "зроби щоб"]):
        level = "FIX_REQUEST"
    elif any(w in user_text for w in ["зовсім", "все не те", "не те зовсім"]):
        level = "TOTAL_REJECT"
    else:
        # LLM-based classification для складних випадків
        level = llm_classify_feedback(user_text)

    # Дія
    if level == "ACCEPTED":
        save_success_pattern()
        return "✓ Патерн збережено. Проект завершено."

    elif level == "WRONG_APPROACH":
        save_error("wrong_method", user_text)
        restart_project(method="Rada+Fabryka")
        return "↻ Перезапускаю з Rada+Fabryka. Рада проаналізує помилку."

    elif level == "FIX_REQUEST":
        extract_fix_instruction(user_text)
        spawn_agent(role="Fullstack", task=user_text)
        return "↻ Запускаю Solo fix за твоїм описом."

    elif level == "TOTAL_REJECT":
        save_error("total_reject", user_text)
        clarification = ask_user("Опиши, що саме не так. Я спробую інакше.")
        update_project_request(clarification)
        restart_project(method="Rada+Fabryka")
        return "↻ Дякую. Перезапускаю з новим розумінням."
```

15.3. Фреймворк якості ТЗ (Spec Quality Framework)

  Цей фреймворк описує як Рада створює якісне ТЗ.
  Він буде оптимізуватись навчанням і досвідом — успішні
  патерни посилюватимуться, помилки виправлятимуться.

  КОЛИ ВИКОРИСТОВУЄТЬСЯ:
  - Кожен Council phase (Rada+Fabryka, Agile Team)
  - Mentor review перед gate
  - Як основа для створення скілу "spec-writing"

  ЩО ВХОДИТЬ:

  1. Секції ТЗ (обов'язково 8+):
     ┌──────────────────────┬──────────────────────────────────┐
     │ Секція               │ Вміст                           │
     ├──────────────────────┼──────────────────────────────────┤
     │ Загальний опис       │ Концепція, цільова аудиторія     │
     │ Технічний стек       │ Технології, архітектура, схема   │
     │ UI/UX                │ ASCII схеми екранів, стани       │
     │ Компоненти/модулі    │ Список, призначення, взаємодія   │
     │ Дані та схеми        │ JSON/YAML приклади, структура    │
     │ Приклади (user story)│ "Як гравець, я хочу..."          │
     │ Ризики               │ Що може піти не так             │
     │ Критерії приймання   | Конкретні, тестовані             │
     └──────────────────────┴──────────────────────────────────┘

  2. Приклади:
     - Кожен модуль має приклад використання
     - User story: "користувач натискає X → бачить Y"
     - JSON/YAML приклади даних з реальними значеннями
     - Код приклади (HTML/CSS/JS) де доречно

  3. Інтерфейс:
     - ASCII схема кожного екрану або стану
     - Контрольний список UI елементів (як Додаток A)
     - Клавіші, жести, touch-взаємодії
     - Стани: пустий, завантаження, помилка, крайні випадки

  4. Якість (що НЕ МОЖНА):
     - Жодних "TODO", "TBD", "далі буде", "якось потім"
     - Жодних "швидко", "зручно", "гарно" без конкретики
     - Конкретні числа: "<3с", "≥44px", не "швидко"
     - Критерії приймання тестовані: "curl → 200", не "працює"

  5. Протиріччя:
     - Однакові терміни всюди (не "кнопка" в секції 2 і "button" в 5)
     - Економічні одиниці узгоджені (якщо є економіка)
     - Секції не суперечать одна одній
     - Roadmap не суперечить поточним можливостям

  6. User stories (обов'язково):
     Формат: "Як <роль>, я хочу <дія>, щоб <мета>"
     Приклад: "Як гравець, я хочу бачити рахунок, щоб
               знати свій прогрес"
     Вимоги:
     - Покривають всі ключові функції
     - Є щасливий шлях (happy path)
     - Є крайні випадки (нічого не знайдено, помилка)

  7. Зовнішнє рев'ю (Expert Review):
     Після створення ТЗ Mentor може відправити його великій
     моделі (Qwen3-235B) з промптом:

     "Ти — Senior Architect. Проаналізуй це ТЗ.
      Знайди МІНІМУМ 5 конкретних проблем:
      | # | Категорія | Секція | Що знайдено | Recommendation |
      Проблеми можуть бути: нереалістичні вимоги, протиріччя,
      пропущені компоненти, невірний стек, відсутні criteria."

     Отримані проблеми → раунд виправлення.
     Як ми робили це ТЗ: v1 → модель знайшла 8 проблем → v3.

  8. Ітерації (Spec Review Cycle):
     v1: Architect пише ТЗ → Mentor review
     Якщо Mentor знайшов >3 проблем: v2 з виправленнями
     Якщо >3 знову: v3
     Максимум 3 ітерації
     Фінальна версія → затвердження → Factory phase

  ОПТИМІЗАЦІЯ ЧЕРЕЗ НАВЧАННЯ:

  Після кожного проекту Learning Loop аналізує:

  │ Метрика                          │ Як впливає                    │
  │──────────────────────────────────┼──────────────────────────────│
  │ Секції, які часто пропускають   │ Додаються до checklist        │
  │ Типові помилки в ТЗ             │ prompt_modifier для ролей     │
  │ User stories, що знадобились    │ Стають шаблоном               │
  │ Критерії, які не спрацювали     │ Виправляються                 │
  │ Зовнішнє рев'ю: часті проблеми  │ Додаються до перевірок Mentor │
  │ Співвідношення часу Council/    │ Оптимізація розподілу часу    │
  │ Factory                         │                              │

  Фреймворк зберігається окремо для легкого оновлення:
  .hermes/frameworks/spec-quality.yaml
  Може бути завантажений як скіл через skill_view()

  ЯК RETRO AGENT ПОКРАЩУЄ СИСТЕМУ:

  З кожною ітерацією Retro Agent вчиться:

  │ Разів │ Дія Retro Agent                               │
  │───────┼───────────────────────────────────────────────│
  │ 1     │ Просто звіт. Без змін.                        │
  │ 3     │ "Developer 3 рази забув тести → prompt_fix"   │
  │ 5     │ "Рада завжди досягає консенсусу за 2 раунди   │
  │       │  → зменшити max_rounds до 3"                  │

  │       │  react-starter"                               │
  │ 20    │ "Загальний тренд: час зменшується,            │
  │       │  якість зростає → система еволюціонує"        │

  Retro Agent може запропонувати:
  - prompt_modifier для конкретної ролі (якщо помилка 3+ рази)
  - Новий скіл (якщо патерн спрацював 3+ рази)
  - Зміну timeout або інших параметрів
  - Issue в GitHub репозиторій СКВА
  - PR зі змінами скілів

  Retro Agent — це окремий Hermes (не Оркестратор), тому:
  - Він не блокує наступний проект
  - Працює в фоні, паралельно
  - Може запускатись навіть якщо Оркестратор вже зайнятий
  - Результати зберігаються в .hermes/retro/
  - Оркестратор читає recommendations перед наступним Council

  КОНТЕКСТ РОЗВИТКУ (розвитку контекст)

  Retro Agent створює і підтримує **контекст розвитку** —
  структуроване сховище досвіду, яке виживає після збою
  сессії і дозволяє легко підхопити з місця переривання.

  Структура контексту:

  .hermes/context/
  ├── development.yaml              # Головний контекст розвитку
  │   # Версія контексту, часом останнього оновлення
  │   # Список всіх проектів з метриками
  │   # Поточний стан Retro Agent (running/done/checkpoint)
  │   # checkpoint: {stage, project, last_analysis_at}
  │
  ├── sessions/
  │   ├── current.checkpoint       # Точка відновлення сессії
  │   │   stage: "analyzing_phases" # Що саме аналізував
  │   │   project: "tetris"         # Який проект
  │   │   progress: 0.6             # 60% готовності
  │   │   started_at: 1717000000    # Коли почав
  │   │   heartbeat: 1717000120     # Останній heartbeat
  │   │
  │   ├── history/                  # Історія сессій Retro Agent
  │   │   ├── 2026-06-07_retro-tetris.yaml
  │   │   └── 2026-06-08_retro-calc.yaml
  │   │
  │   └── resume/                   # Точки для підхвату
  │       └── 2026-06-07.checkpoint
  │           # Містить: стан аналізу, проміжні результати,
  │           # список файлів які вже прочитав
  │
  └── readiness/                    # Готовність системи
      ├── skills-maturity.yaml      # Які скіли дозріли
      ├── agents-experience.yaml    # Досвід кожної ролі
      └── system-health.yaml        # Стан здоров'я системи

  Механізм checkpoint (якщо сессія збилась):

  1. Retro Agent кожні 30с зберігає checkpoint:
     .hermes/context/sessions/current.checkpoint

     ```yaml
     stage: "analyzing_phases"
     project: "tetris"
     progress: 0.6
     files_analyzed:
       - .hermes/phases/history/01-council/gate.yaml
       - .hermes/phases/history/02-factory/gate.yaml
     pending:
       - .hermes/phases/history/03-deploy/gate.yaml
     partial_results:
       council_duration: 65s
       factory_duration: 140s
       deploy_duration: null  # ще не проаналізовано
     ```

  2. При краху сессії: checkpoint залишається на диску

  3. При наступному запуску Retro Agent:
     - Виявляє current.checkpoint
     - Читає stage, progress, pending
     - Продовжує з місця переривання
     - Не починає з нуля

  4. Після завершення:
     - checkpoint видаляється
     - Результат записується в .hermes/retro/{project}/
     - Контекст оновлюється: development.yaml

16.2. Правила

1. **Спочатку дешевше:** завжди пробувати найдешевшу модель
2. **Ескалація при fail:** якщо верифікація не пройшла → підвищити
3. **Деградація при retry:** при повторній спробі → зменшити модель
4. **Бюджет проекту:** ліміт на проект в config.yaml
5. **Попередження:** при 80% бюджету → повідомлення
6. **Стоп:** при 100% → зупинка + пропозиція спростити

```yaml
# .hermes/project/config.yaml
budget:
  max_cost: 100           # умовні одиниці
  model_selection: auto   # auto | flash-only | premium
  warn_at: 80             # % при якому попередити
  stop_at: 100            # % при якому зупинитись
```

  16.3.7. Credential Pool — API ключі як ресурс

  Користувач може надати декілька API ключів. Система
  використовує їх як пул — більше ключів = більше
  паралельних викликів = більше агентів одночасно.

  Як це працює в Resource Balancing:

  ```python
  def calculate_capacity(machine):
      # Фізичні ресурси (CPU, RAM)
      from_cores = cpu_cores - load_avg
      from_ram = free_ram // 2

      # API ресурси (скільки паралельних викликів дозволяють ключі)
      api_slots = 0
      for provider, keys in credential_pool.items():
          for key in keys:
              rate = key.rate_limit  # наприклад, 5 RPM для DeepSeek
              api_slots += rate

      # Ліміт з registry
      from_registry = machine.max_agents

      # Capacity = мінімум з усіх лімітів
      max_new = min(from_cores, from_ram, api_slots, from_registry)
      max_new = max(max_new, 1)

      return max_new
  ```

  Приклад:

  ```
  Користувач надав:
  - 2 ключі DeepSeek (кожен 5 RPM)  → 10 слотів
  - 1 ключ Qwen3-235B (2 RPM)       → 2 слоти
  - 1 ключ GPT-4o (3 RPM)           → 3 слоти
  Всього API слотів: 15

  Комп'ютер: 4 ядра, 8GB RAM → 4 слоти
  Реальна capacity: min(4, 15) = 4 агенти
  ```

  Якщо додати ще один комп'ютер (Hetzner, 8 ядер):

  ```
  WSL:    4 ядра  → 4 агенти
  Hetzner: 8 ядер  → 8 агентів
  API:              → 15 слотів
  Всього: 12 агентів на 2 машинах
  ```

  Credential pool в config:

  ```yaml
  # ~/.hermes-skva/credentials.yaml
  providers:
    deepseek:
      keys:
        - key: "sk-xxx1"
          rate_limit: 5  # RPM
        - key: "sk-xxx2"
          rate_limit: 5
    gonka:
      keys:
        - key: "gsk-xxx"
          rate_limit: 2
          models: ["qwen3-235b"]
  ```

17.2. SSH spawn механізм

```python
def spawn_via_ssh(machine, role, task, remote_project_path):
    """Запустити Hermes агента на віддаленій машині через SSH."""

    prompt = f"""
Твоя роль: {role.name}
Скіли: {role.skills}

Проект: {remote_project_path}
Задача: {task}

Працюй в .hermes/artifacts/{role.name}/
Heartbeat: .hermes/signals/heartbeat/{role.name}.live кожні 60с
По завершенню: touch .hermes/signals/.{phase}.done
Не чіпай файли інших агентів
Git push після кожного артефакту
"""

    # Екранувати для shell-in-shell
    safe_prompt = prompt.replace('"', '\\"').replace("'", "'\\''")

    ssh_command = (
        f"ssh -i {machine.key} -o ConnectTimeout=10 "
        f"{machine.user}@{machine.host} "
        f"'cd {remote_project_path} && "
        f"source ~/.hermes/.env && "
        f"hermes chat -q \"{safe_prompt}\"'"
    )

    terminal(
        background=True,
        command=ssh_command,
        timeout=1800
    )
```

17.3. Git синхронізація

1. Оркестратор ініціалізує проект з Git репозиторієм
2. Кожен агент має доступ до Git (локально або через SSH)
3. Агенти не створюють окремі бранчі — вони пишуть
   в ізольовані директорії .hermes/artifacts/<role>/
4. Після кожного артефакту: git add → git commit → git push
5. Оркестратор merge артефакти в src/ після gate

Конфлікти:
- При ізоляції по директоріях конфлікти неможливі
- Якщо два агенти пишуть в один файл (наприклад, два Dev
  редагують package.json) — використовується lock-файл
- Lock: .hermes/signals/locks/artifacts.lock (flock)
- Якщо конфлікт виник — Merge Agent (Hermes з роллю Merger)

17.4. SSH безпека

- Використовувати SSH ключі (не паролі)
- Ключі зберігаються в ~/.ssh/ (стандартне місце)
- Обмеження: ключі мають права 600 (тільки власник)
- Рекомендація: створити окремого користувача для агентів
  (не root), але поточна інфраструктура використовує root
  — це прийнятно для ізольованого сервера.

20.2. Rollback (відкат до робочої версії)

Кожна зміна СКВА версіонується через Git:

```yaml
# .hermes-skva/CHANGELOG.yaml
versions:
  - version: "1.0.0"
    date: "2026-06-01"
    cambios: "Initial release"
    stable: true

  - version: "1.1.0"
    date: "2026-06-15"
    changes: "Fix heartbeat direction"
    parent: "1.0.0"
    stable: true

  - version: "1.2.0-beta"
    date: "2026-06-20"
    changes: "Add vector memory (experimental)"
    parent: "1.1.0"
    stable: false  # beta — не для всіх

  - version: "1.1.0"  # ← rollback сюди
    rollback_from: "1.2.0-beta"
    reason: "vector memory caused 3x API costs"
    stable: true
```

Механізм rollback:

```python
def rollback_skva(version="1.1.0"):
    """Відкотити скіли СКВА до вказаної версії."""
    # 1. Зберегти поточну версію як бекап
    shutil.copytree(hermes_home / "skills", hermes_home / "skills.bak")

    # 2. Замінити скіли на версію з Git
    subprocess.run(["git", "checkout", f"v{version}", "--", "skills/"])

    # 3. Перевірити що базові Acceptance Criteria проходять
    test_result = run_acceptance_tests()
    if test_result.failed:
        # Відкатити бекап
        shutil.copytree(hermes_home / "skills.bak", hermes_home / "skills")
        return f"ROLLBACK FAILED: {test_result.errors}"

    # 4. Оновити CHANGELOG
    append_to_changelog(rollback_to=version)

    return f"Rolled back to v{version}"
```

Користувач:

```
> /skva version
  Поточна: v1.2.0-beta (нестабільна)
  Стабільна: v1.1.0

> /skva rollback 1.1.0
  ✓ Відкочено до v1.1.0
  ✓ Acceptance Criteria passed

> /skva version
  Поточна: v1.1.0 (стабільна)
```

```python
def suggest_pr(skill_path):
    """Запропонувати створення Pull Request в основний репозиторій."""

    # 1. Аналіз: чи є такий скіл в основній версії?
    if skill_exists_upstream(skill_path):

    if not test_compatibility(skill_path):

    generate_docs(skill_path)

        title=f"feat: add {skill_name}",
```

21.6. Безпека

  Політика безпеки для СКВА:

  1. SSH ключі: тільки restricted user (не root в ідеалі)
  2. GitHub токен: мінімальні права (repo + workflow)
  3. Prompt injection: екранувати user-request.md перед вставкою в prompt
  4. Логи: не містять API ключів (автоматичне маскування)
  5. Тестові проекти: в ізольованій директорії /tmp/skva-*
  6. Git push: тільки в дозволені репозиторії
  7. Оновлення: перевірка підпису (або HTTPS, не git://)

D.4. НАЛАШТУВАННЯ AGENT REGISTRY (якщо є інші машини)

Файл: `~/.hermes-skva/agents/registry.yaml`

```yaml
# Мінімальний варіант (тільки цей комп'ютер):
machines:
  - name: local
    transport: terminal
    host: localhost
    max_agents: 3
    roles:
      - Fullstack
      - Developer
      - Architect
    models:
      - deepseek-flash
      - deepseek
      - qwen3-235b

# З віддаленим сервером:
  - name: my-server
    transport: ssh
    host: 192.168.1.100        # або IP вашого сервера
    user: root                  # або ваш username
    key: ~/.ssh/id_rsa         # шлях до SSH ключа
    max_agents: 2
    roles:
      - Developer
      - QA
    models:
      - qwen3-235b
```

**SSH ключ — як створити (якщо ще нема):**
```bash
ssh-keygen -t ed25519 -C "skva-agent"
# → ~/.ssh/id_ed25519 (закритий) + ~/.ssh/id_ed25519.pub (відкритий)

# Скопіювати на сервер:
ssh-copy-id user@server-ip
# Або вручну: додати в ~/.ssh/authorized_keys на сервері

# Перевірити:
ssh user@server-ip "echo connected"
# → connected
```

**GitHub токен — як створити (для Git синхронізації):**
1. Відкрий https://github.com/settings/tokens
2. Generate new token → Classic
3. Вибери: repo, workflow
4. Скопіюй токен
5. Додай в ~/.hermes/.env:
```bash
echo "GITHUB_TOKEN=ghp_ваш_токен" >> ~/.hermes/.env
```

D.6. ТИПОВІ ПРОБЛЕМИ ТА РІШЕННЯ

| Симптом                      | Причина                         | Що робити                          |
|──────────────────────────────┼─────────────────────────────────┼────────────────────────────────────│
| "command not found: hermes"  | Hermes не встановлено           | `curl -fsSL ...install.sh \| bash`  |
|                              | або не в PATH                   | `source ~/.bashrc`                  |
| "skill not found: skva-..."  | СКВА не встановлено             | `hermes skills install ...`         |
|                              |                                 | або перевір `hermes skills list`    |
| "SSH connection refused"     | SSH сервер не запущено          | `sudo systemctl start ssh`          |
|                              | на віддаленій машині            | або перевір IP: `ping server-ip`   |
| "SSH key not found"          | Неправильний шлях до ключа     | Перевір `registry.yaml` key: шлях  |
|                              |                                 | `ls ~/.ssh/` — чи є файл           |
| "Git push failed"            | Немає GitHub токену             | Додай GITHUB_TOKEN в ~/.hermes/.env |
|                              | або токен недійсний             | Перевір: `gh auth status`           |
| "Git: не вказано remote"     | Проект не прив'язаний до Git    | `git remote add origin <url>`       |
| "No model available"         | Провайдер не налаштовано        | `hermes model` — вибрати модель     |
| "Agent timed out"            | Задача занадто складна для     | Збільшити timeout в config.yaml    |
|                              | поточного методу                | або змінити метод                   |
| "Heartbeat missing"          | Агент впав або машина          | Оркестратор автоматично retry×3    |
|                              | перевантажена                   | Потім заміна агента                 |
